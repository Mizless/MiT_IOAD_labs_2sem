# БРИФИНГ ПРОЕКТА: AI-бот для игры сбора сокровищ
# Версия для передачи следующему ИИ

================================================================================
## 1. ЧТО ЗА ИГРА
================================================================================

TCP-сервер (Unity) на 127.0.0.1:8080. Две команды (red / blue) по 5 агентов.
Цель: собирать сокровища с карты и доставлять на свою базу за 10 минут.
Клиент пишется на Python, общается по JSON через TCP.

Протокол:
- Клиент → {"team": "red"}
- Сервер → {"type": "joinAccepted", "playerId": "...", "team": "red"}
- Клиент → {"actions": [{"action": "ready"}]}
- Сервер шлёт снапшоты 10 раз/сек (каждые 0.1с):
  {"tick":N, "gameTime":T, "gameState":"InGame", "bases":[], "agents":[], "treasures":[], "golems":[]}
- Клиент → {"actions": [{"Id": "agentId", "action": "position", "target": {x,y,z}}, ...]}

## 2. ПРАВИЛА ИГРЫ (ТОЧНЫЕ ЦИФРЫ)
================================================================================

АГЕНТЫ:
- Каждая команда: 5 агентов
- Базовая скорость: 5.0 м/с (без груза)
- Скорость с грузом: 5 * (1 - 0.005 * weight) м/с
  → Груз 60кг  = скорость 3.5 м/с (голем В ПРЕСЛЕДОВАНИИ догоняет!)
  → Груз 100кг = скорость 2.5 м/с (голем ПАТРУЛЬНЫЙ тоже догоняет!)
  → Груз 200кг = скорость 0.5 м/с (минимум)
- Инвентарь: 1 слот (одно сокровище любого веса)

СОКРОВИЩА:
- Характеристики: weight (вес, кг), value (ценность, очки)
- Подобрать: команда "pickup" когда в зоне подбора
- Сбросить: команда "drop" — сокровище падает позади агента
- Доставка: войти в радиус 2м от своей базы и дропнуть → очки = value

ГОЛЕМЫ (нейтральные стражи):
- Скорость патруля:     2.0 м/с
- Скорость погони:      3.5 м/с
- Угол обзора:          90°
- Дальность обнаружения: 10 м
- Дальность потери:     15 м
- Маршрут:              4 контрольные точки, замкнутый
- При контакте с агентом: оглушение 5 сек + выброс груза
- После атаки: голем игнорирует всех 5 сек, возвращается к патрулю

ОГЛУШЕНИЕ (Stun):
- Причины: контакт с големом, успешная кража противником
- Длительность: 5 секунд
- Эффект: агент не может делать ничего, груз выпадает
- После оглушения: агент "забывает" цель (сбрасывается NavMesh)

КРАЖА (Steal):
- Заряжается: только когда тележка пуста, 50 секунд на полный заряд
- Условие применения: вор в радиусе 1м от жертвы, тележка вора пуста
- Эффект: вор получает груз жертвы + ускорение 5 сек (игнорирует вес)
  Жертва оглушается на 5 сек
- Поле в снапшоте: agentState.stealAbilityReady (bool)

БАЗА:
- Радиус приёма: 2 метра
- Статичная, не блокирует движение
- У каждой команды своя (красная/синяя)

## 3. ПРОТОКОЛ — ТОЧНЫЙ ФОРМАТ КОМАНД
================================================================================

ВСЕ КОМАНДЫ ТРЕБУЮТ ПОЛЕ target (кроме drop и steal — но лучше давать):
  "position": ОБЯЗАТЕЛЬНО target={x,y,z}
  "pickup":   ОБЯЗАТЕЛЬНО target={x,y,z} (координаты сокровища)
  "drop":     target={x,y,z} — можно передавать, сервер не ломается
  "steal":    target={x,y,z} — можно передавать, сервер не ломается
  "ready":    без target (только в лобби)

BATCH-формат:
{"actions": [
    {"Id": "-12385", "action": "position", "target": {"x": 10, "y": 0, "z": 5}},
    {"Id": "-9566",  "action": "pickup",   "target": {"x": 3,  "y": 0, "z": 9}}
]}

ВАЖНО: Id агента — это agentId из снапшота, НЕ playerId.

Снапшот агента:
{"agentId": "-12385", "team": "red", "pos": {x,y,z},
 "hasTreasure": bool, "isStunned": bool,
 "stealAbilityReady": bool (опционально),
 "currentSpeed": float (опционально)}

Снапшот сокровища:
{"id": "4521", "pos": {x,y,z}, "isPicked": bool,
 "holderAgentId": str|null (опционально),
 "weight": float, "value": float}

## 4. КРИТИЧЕСКИЕ БАГИ КОТОРЫЕ ЛОМАЛИ КОД
================================================================================

БАГ №1 — УБИЙСТВЕННЫЙ: Логирование внутри _cmd()
  Проблема: открытие файла на каждую команду (file I/O) сдвигает тайминги.
  Симптом: агенты доходят до монеты и встают АФК — pickup не отправляется.
  Правило: НИКОГДА не делать I/O внутри _cmd() или decide().
  Решение: логировать только в главном цикле send_msg, после отправки.

БАГ №2: порог drop у базы = 0.5 вместо 1.9
  Проблема: бот упирается в коллайдер базы на ~1.0-1.5м, не может подойти.
  Симптом: агент стоит у базы с монетой вечно (drop никогда не вызывается).
  Правило: BASE_DROP_RADIUS = 1.9 — ПРОВЕРЕНО, НЕ МЕНЯТЬ.

БАГ №3: pickup без target
  Проблема: сервер игнорирует pickup без target.
  Симптом: агент стоит над монетой АФК.
  Правило: pickup ВСЕГДА с target = координаты сокровища.

БАГ №4: изменение сигнатуры _cmd(action, target) → _cmd(action, target, team_name)
  Проблема: если хоть один вызов _cmd не передаёт team_name — TypeError, бот падает.
  Симптом: тихий крэш или АФК (socket закрывается).
  Правило: НЕ добавлять параметры в _cmd.

БАГ №5: команда ready с Id агента
  Проблема: ready отправляется до того как известны agentId.
  Правило: {"actions": [{"action": "ready"}]} — без Id.

## 5. СТРУКТУРА ТЕКУЩЕГО РАБОЧЕГО КОДА (client_red_smart.py)
================================================================================

ГЛОБАЛЬНЫЕ КОНСТАНТЫ:
  AGENT_BASE_SPEED   = 5.0    # м/с без груза
  GOLEM_CHASE_SPEED  = 3.5    # м/с голем в преследовании
  GOLEM_PATROL_SPEED = 2.0    # м/с голем патрулирует
  GOLEM_VISION_DIST  = 10.0   # м дальность обнаружения
  GOLEM_LOSE_DIST    = 15.0   # м дистанция потери
  STEAL_RADIUS       = 1.0    # м радиус кражи
  BASE_DROP_RADIUS   = 1.9    # м — ПРОВЕРЕНО, НЕ МЕНЯТЬ
  CATCHABLE_WEIGHT   = 60     # кг — при этом весе голем догоняет

ГЛОБАЛЬНАЯ ПАМЯТЬ (team_memory):
  "treasure_map"   : dict[id → treasure]   — карта всех известных сокровищ
  "base_pos"       : {x,y,z}|None          — наша база
  "enemy_base_pos" : {x,y,z}|None          — база врага
  "golem_prev_pos" : dict[id → {x,y,z}]   — прошлая позиция голема
  "golem_dir"      : dict[id → {x,z}]     — вектор движения голема (из разности позиций)
  "prev_points"    : int                   — предыдущий счёт (для детекта доставки)
  "delivery_count" : int                   — счётчик доставок
  "assignments"    : dict[treasure_id → agent_id] — жёсткое распределение

КЛАСС Agent (на агента):
  Поля:
    id             — строковый agentId
    last_cmd_time  — время последней отправленной команды (rate-limit)
    cmd_interval   = 0.08 сек (rate-limit: не спамим сервер)
    last_sent_target, last_sent_time — для throttle position (не спамим одну точку)
    pos_hist       — история позиций для детекта застревания (последние 20)
    unstuck_target, unstuck_until — рывок при застревании
    explore_index  — индекс точки исследования карты
    carried_weight — вес текущего груза (запоминается при выборе цели)

  Методы:
    _cmd(action, target=None) — единственный способ отправить команду.
      Внутри: throttle для position + rate-limit 0.08сек.
      НЕ МЕНЯТЬ СИГНАТУРУ.

    _my_assignment() → treasure_id|None
      Возвращает ID монеты назначенной этому агенту.

    _release_assignment()
      Освобождает назначение агента (при оглушении, доставке, дропе).

    _assess_golems(pos, golems, has_treasure, weight) → (golem|None, threat_level)
      threat_level:
        0 = нет угрозы
        1 = осторожно (уклониться если можем)
        2 = срочный побег
        3 = бросать груз (голем точно догонит)
      Учитывает: вектор движения голема, state="Chase", вес груза.

    _escape_direction(pos, gpos, base_pos) → {x,y,z}
      Умный побег: если база в той же стороне что и "от голема" — бежим к базе.
      Иначе — от голема на 18м.

    decide(me, treasures, bases, game_state, team, golems, all_agents) → cmd|None
      Главная логика. Порядок приоритетов (важно!):
        1. Обновить карту сокровищ
        2. Выйти если не InGame
        3. Обработать оглушение
        4. Найти базы
        5. Anti-stuck
        6. Оценка голема → действие по threat_level
        7. Доставка если есть груз
        8. Кража если stealAbilityReady
        9. Сбор сокровищ с жёстким назначением

ПОРЯДОК ОБРАБОТКИ АГЕНТОВ:
  my_agents.sort(key=lambda a: 0 if a.get("hasTreasure") else 1)
  Носители груза обрабатываются ПЕРВЫМИ — они резервируют команды раньше.

## 6. ЛОГИКА ИССЛЕДОВАНИЯ КАРТЫ
================================================================================

13 точек по карте (углы, середины сторон, центр, промежуточные):
  (±80, 0, ±80), (±80, 0, 0), (0, 0, ±80), (0,0,0),
  (±40, 0, ±40)

Агент переходит к следующей точке когда dist < 3м.
explore_index у каждого агента свой → они расходятся по карте.

================================================================================
## 7. ЛОГИКА РАСПРЕДЕЛЕНИЯ СОКРОВИЩ (assignments)
================================================================================

team_memory["assignments"] = {treasure_id: agent_id}

Правила:
- Если сокровище свободно → назначить себе, освободить старое назначение
- Если сокровище назначено другому → искать альтернативу среди свободных
- В score(): своё назначение ×2.5, чужое ×0.002
- Очищается: при исчезновении сокровища (подобрано / вышло из видимости)
- Очищается: при оглушении агента (_release_assignment)
- Очищается: при доставке (_release_assignment)
- Очищается: при drop из-за голема

Проблема: assignments НЕ синхронизируется между агентами атомарно.
При обработке агентов в одном тике первый агент может назначить себе монету,
которую второй ещё не видел как занятую.
Порядок обработки (носители первыми) частично решает это.

================================================================================
## 8. ФОРМУЛА SCORE (оценка привлекательности сокровища)
================================================================================

d_to   = dist(агент → сокровище)
d_home = dist(сокровище → база)
spd    = speed_with_weight(weight)
speed_factor = spd / 5.0  # нормировка: 1.0 = без груза

total_cost = (d_to + d_home) / speed_factor   # время доставки (с учётом замедления)
s = value / total_cost                          # эффективность = очки/время

Модификаторы:
  × 2.5  если это наше назначенное сокровище
  × 0.002 если назначено другому агенту
  × 0.25 если is_catchable(weight) И голем ближе 12м
  × 80.0 если d_to < 2.0 (почти рядом — добираем в первую очередь)

## 9. СЛАБЫЕ МЕСТА ТЕКУЩЕГО КОДА
================================================================================

СЛАБОЕ МЕСТО №1: carried_weight — вес угадывается заранее, не из стейта
  Проблема: агент запоминает вес сокровища ДО подбора (при выборе цели).
  Если агент был переназначен или оглушён — carried_weight может быть неверным.
  Сервер не возвращает вес в данных агента (только в данных сокровища).
  Идеальное решение: при подтверждении hasTreasure=true → найти сокровище
  которое исчезло из available рядом с агентом и запомнить его вес.

СЛАБОЕ МЕСТО №2: кража — нет предсказания перехвата
  Текущий код просто идёт к текущей позиции врага.
  Враг движется → агент догоняет хвост.
  Улучшение: предсказывать позицию врага через T секунд и идти туда.
  Формула: intercept = enemy_pos + enemy_velocity * (dist/our_speed)

СЛАБОЕ МЕСТО №3: нет блокировки голема союзниками
  Правила говорят: пустой союзник может встать между соратником с грузом и
  Geometridae, перекрыв угол зрения 90°. Это сильная тактика для тяжёлых грузов.
  В коде не реализовано вообще.

СЛАБОЕ МЕСТО №4: assignments не персистентны между тиками при гонках
  Если два агента обрабатываются в одном тике и оба видят свободную монету —
  первый берёт, второй должен уйти. Но если у обоих score() выбирает одну монету,
  второй получит её в assignments только если нашёл альтернативу.
  Текущее решение: sort (носители первыми) + штраф 0.002. Работает, но не идеально.

СЛАБОЕ МЕСТО №5: golem_dir — вектор из разности позиций, не нормирован
  Текущий код хранит "сырую" разность позиций голема между тиками.
  При маленьких движениях (голем поворачивает) — вектор нестабилен.
  Улучшение: нормировать вектор и использовать EMA (скользящее среднее).

СЛАБОЕ МЕСТО №6: голем "Patrol" пугает нас с 22м если несём тяжёлый груз
  Текущий порог panic_dist=22 для catchable груза.
  Это может заставлять агентов бежать от патрульного голема который их не видит.
  Голем видит только на 10м — при дальности 22м он нас ещё не замечает.
  Улучшение: разделить логику для "Patrol" (реагировать только на 12м) и
  "Chase" (реагировать на 22м).

СЛАБОЕ МЕСТО №7: нет детекта "голем атакует союзника"
  Если голем погнался за союзником — наш агент не знает и может случайно
  оказаться на пути голема.

СЛАБОЕ МЕСТО №8: explore_index не координируется между агентами
  Все агенты начинают с explore_index=0,1,2,3,4 (через agent_index).
  Но при рестарте (clear agents_ai) индекс сбрасывается.
  Улучшение: распределять explore_points между агентами явно.

СЛАБОЕ МЕСТО №9: steal priority formula может гнать за слабой целью
  Формула: (1/(ed+0.5)) * (1 + eb_dist/20)
  Враг далеко от своей базы (eb_dist велик) → высокий приоритет.
  Но если этот враг далеко от нас — мы теряем время погони.
  Лучше: учитывать наше время до перехвата vs время врага до базы.

СЛАБОЕ МЕСТО №10: нет реакции на "конец сокровищ"
  Когда все сокровища собраны или недоступны — агенты исследуют карту.
  Но можно добавить: если steal заряжен → сфокусировать всех воров на врагах.
